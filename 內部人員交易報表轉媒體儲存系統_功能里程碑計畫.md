# 內部人員交易報表轉媒體儲存系統

## 功能里程碑計畫 v1.1

## 0. 既有系統覆蓋前提

本計畫以前公司內已存在舊系統為前提。新系統功能開發不得無視舊系統既有功能、報表邏輯、資料欄位、權限規則、排程流程、匯出格式與使用者操作習慣。

本文件只列出可拆任務卡的功能里程碑、功能描述與驗收條件；系統架構、資料流程、權限模型、PDF 儲存與浮水印策略，應以同目錄的「系統架構與治理計畫書」為準。分階段演練、三人小隊責任矩陣、validators、test cases、evidence 與 Gate，應以 `drills/分階段演練與驗收計畫.md` 為準。

任務卡派工、Agent role、reviewer、validator、human sign-off、ADR gate 與違規阻擋機制，應依《內部人員交易報表轉媒體儲存系統_Agent Team計畫書》v1.0 執行；本文件只定義功能要做什麼，不取代 Agent Team 協作規則。

每一個功能里程碑都必須先完成舊系統盤點，再決定採用以下方式之一：

| 處理方式 | 適用情境 |
| --- | --- |
| 直接沿用 | 舊邏輯正確、風險低、可封裝使用。 |
| 代碼複製 / 移植 | 舊邏輯複雜但結果可靠，需先保留一致性。 |
| 重寫 | 舊代碼不可維護、效能不足、耦合過高或無法跨平台。 |
| 重寫並優化 | 需維持舊結果一致，但改善架構、效能、安全性或可測試性。 |

任何重寫項目都必須有舊系統對照、測試資料、預期結果與差異比對紀錄。新系統正式取代前，必須證明能涵蓋舊系統必要功能範圍。

遷移階段對應如下：

| 遷移階段 | 功能里程碑 |
| --- | --- |
| Phase 0：盤點、PoC 與測試基準 | M0 |
| Phase 1：Shadow DB、Staging 與稽核基礎 | M1、M9-01 |
| Phase 2：檢核、報表、PDF 與下載閘道 | M2、M3、M4、M5 |
| Phase 3：查詢、覆核、Admin 與營運稽核 | M6、M7、M8 |
| Phase 4：Linux / Container 相容驗證 | M9-02、M9-03 |
| Phase 5：Pilot、分批切換與正式驗收 | M9-04、M9-05、M10 |

本表是唯一 Phase↔M 功能對照表；若演練階段、任務卡 Stage 或 Gate 調整，需同步更新 `drills/分階段演練與驗收計畫.md` 與 `tasks/README.md`，不得在其他文件另建平行對照表。

## 1. 里程碑 M0：舊系統盤點與功能基準

### M0-01 盤點既有報表

功能描述：

* 盤點舊系統所有內部人員交易相關報表。
* 記錄報表代號、報表名稱、輸出格式、欄位、排序、篩選條件、群組方式與彙總規則。
* 標記正式使用、停用、低使用率與待確認報表。

驗收條件：

* 完成舊報表清單。
* 每份正式報表都有欄位與產製邏輯摘要。
* 每份正式報表都有新系統承接方式：沿用、移植、重寫或廢止候選。

### M0-02 盤點既有資料來源

功能描述：

* 盤點交易資料、人事資料、部門資料、內部人名單、限制名單、觀察名單與 AD / 群組來源。
* 記錄來源系統、資料表、檔案格式、API、排程、欄位定義與更新頻率。
* 記錄現有 stored procedure、view、trigger、batch job 與手動匯入流程。

驗收條件：

* 完成資料來源清單。
* 完成欄位對照表。
* 標記需要移植、重寫或淘汰的 DB 物件。

### M0-03 建立舊系統結果基準

功能描述：

* 選定基準批次、基準資料日期與基準報表。
* 保存舊系統輸出結果、筆數、金額、頁數、PDF metadata 與摘要 hash。
* 建立 Golden Dataset 與 expected result。
* 建立 Golden Dataset 的資料來源、資料血緣、維護責任、核准紀錄與更新流程。
* 建立 Shadow Validation Data 與 Golden Dataset 的區隔規則。

驗收條件：

* Golden Dataset 不含未脫敏正式資料。
* 每個測試案例都有 expected result。
* 舊系統基準輸出可重複取得或可查證。
* Golden Dataset 可用於 SIT、CI 或報表比對，不直接依賴正式資料庫。
* Shadow Validation Data 若涉及受控正式資料驗證，需有隔離、權限、稽核與核准紀錄。

### M0-04 建立第三方與跨平台 PoC

功能描述：

* 驗證 PDF library 在 Linux、中文字型、浮水印、簽章與大量產生情境可用。
* 驗證 Object Storage / NAS 的權限、加密、備份與還原。
* 驗證 AD / OIDC / LDAPS 整合可用。
* 比較 MSSQL on Linux、PostgreSQL、MariaDB 與原 MSSQL 相容性。
* 本演練依 ADR-013 以 MariaDB 作為目標 DB，但正式專案仍需保留完整 DB PoC 與最終選型判斷。

驗收條件：

* 完成 PoC 結果紀錄。
* 標記不可用、需補強或可採用的技術項目。
* DB 選型不得只憑偏好決定，需有相容性與維運依據。

### M0-05 建立 SP / SQL 改造分類

功能描述：

* 將 stored procedure、view、trigger、batch job 分類為保留、封裝、重寫或淘汰。
* 標記 T-SQL、cursor、temp table、transaction、lock、日期函式與排序規則等相容性風險。
* 建立每個 DB 物件的舊系統對照與驗證方式。

驗收條件：

* 每個高風險 DB 物件都有處理策略。
* 重寫項目有預期結果與比對方式。
* 淘汰項目有業務或稽核確認。

### M0-06 建立角色、RACI 與關鍵依賴

功能描述：

* 定義業務、稽核、資安、DBA、QA、DevOps、開發與系統維運責任。
* 標記各里程碑的前置依賴與 critical path。
* 建立問題升級與決策窗口。

驗收條件：

* 每個里程碑都有負責與參與角色。
* 需業務、稽核、資安、DBA 或維運簽核的項目已列出。
* Critical path 可被追蹤。

## 2. 里程碑 M1：資料擷取與 Staging

### M1-01 建立匯入批次管理

功能描述：

* 建立 Batch ID。
* 記錄來源系統、來源檔案、來源筆數、匯入時間、匯入人員或排程 ID。
* 支援匯入狀態：待處理、處理中、成功、失敗、已重跑。

驗收條件：

* 每次匯入都有唯一 Batch ID。
* 匯入失敗可查錯誤原因。
* 批次狀態可追蹤。

### M1-02 建立 Staging 資料表

功能描述：

* 匯入資料先進入 Staging，不直接寫入正式交易資料。
* 保存原始欄位、標準化欄位、來源批次與錯誤欄位。
* 支援重跑同一批次。

驗收條件：

* 重跑不造成重複正式資料。
* Staging 資料可追溯來源批次。
* 原始資料與標準化資料可比對。

### M1-03 移植或重寫舊資料轉換邏輯

功能描述：

* 盤點舊系統資料轉換 SQL、stored procedure 或程式邏輯。
* 對結果可靠的邏輯先移植或封裝。
* 對效能差或不可維護的邏輯重寫並建立比對測試。

驗收條件：

* 新舊轉換結果筆數一致。
* 主要欄位值一致。
* 差異有紀錄與處理結論。

### M1-04 建立稽核寫入基礎

功能描述：

* 建立共用稽核寫入介面。
* 建立 trace ID、correlation ID、actor、action、target、result、reason code 與 risk level 欄位。
* 建立事件 payload 與 payload hash。

驗收條件：

* 匯入、轉換、權限異動與系統參數異動可寫入稽核。
* 稽核寫入失敗時，關鍵操作不可靜默成功。
* reason code、risk level、action、result 使用共同字典。

## 3. 里程碑 M2：資料檢核

### M2-01 建立檢核規則版本

功能描述：

* 建立檢核規則表與規則版本。
* 支援規則啟用、停用、生效時間與異動原因。
* 先承接舊系統既有檢核規則。

驗收條件：

* 每條檢核規則有版本與異動紀錄。
* 舊系統既有檢核規則完成對應。
* 規則異動需被稽核。

### M2-02 建立檢核結果等級

功能描述：

* 支援 Pass、Warning、Error、Critical。
* Warning 可進報表但需標記。
* Error 不可產生正式報表。
* Critical 需中止批次並通知。

驗收條件：

* Golden Dataset 的 Pass / Warning / Error / Critical 結果符合 expected result。
* Critical 事件會產生告警。
* 檢核報告可追溯 Batch ID 與規則版本。

### M2-03 建立新舊檢核比對

功能描述：

* 使用舊系統檢核結果作為基準。
* 比對檢核筆數、錯誤原因、警示分類與批次結果。

驗收條件：

* 新舊檢核結果一致或差異已核准。
* 差異原因可追蹤。
* 差異修正後可重跑驗證。

### M2-04 建立 Data Scope 基礎規則

功能描述：

* 建立使用者、角色、部門、分公司、報表類型與機密等級的資料範圍判斷。
* 建立 Data Scope 測試案例。
* 建立資料範圍異動的稽核寫入。

驗收條件：

* 查詢、預覽、下載與覆核可套用 Data Scope。
* 無權限、跨部門、高機密與例外授權案例可被測試。
* Data Scope 異動有稽核紀錄。

## 4. 里程碑 M3：報表產生與版本管理

### M3-01 建立報表模板版本

功能描述：

* 建立報表模板、模板版本、報表代號與機密等級。
* 盤點並承接舊系統報表版型。
* 模板異動需覆核後生效。

驗收條件：

* 舊系統正式報表都有新模板對應。
* 模板版本可追蹤。
* 模板異動有申請與覆核紀錄。

### M3-02 建立報表產生任務

功能描述：

* 建立 Report Run ID。
* 支援手動產生、排程產生、指定資料日期、指定批次與指定報表代號。
* 報表產生失敗需保存錯誤摘要。

驗收條件：

* 每次產生都有 Report Run ID。
* 產生失敗可查原因。
* 不暴露連線字串、Token 或敏感設定。

### M3-03 建立報表版本與狀態機

功能描述：

* 每份報表需有 Report ID 與 Report Version。
* 支援待產生、產生中、產生成功、待覆核、已覆核、已發布、已歸檔。
* 支援產生失敗、退回重產、已取代、作廢申請中、已作廢。

驗收條件：

* 重產不覆蓋舊版。
* 舊版可標記已取代。
* 已發布報表不可直接刪除。

### M3-04 建立新舊報表雙製比對

功能描述：

* 同一批 Golden Dataset 同時產生舊系統與新系統報表。
* 比對筆數、金額、欄位、排序、頁數、PDF metadata、canonical text hash。
* 驗證 PDF library 可在 Linux Container 產生基準 PDF。
* 比對 Windows 與 Linux Container 產出的頁數、metadata、canonical text、中文字型、浮水印位置與可讀性。

驗收條件：

* 主要報表與 expected result 一致。
* 新舊報表差異有紀錄。
* 差異需經業務或稽核確認。
* Linux Container PDF 產出已建立 baseline；若差異無法解釋，不得進入正式切換 Gate。

## 5. 里程碑 M4：PDF 主檔保存

### M4-01 建立 PDF metadata

功能描述：

* 保存報表代號、資料日期、批次編號、版本、機密等級、模板版本、Report Run ID。
* 保存 PDF 主檔路徑與 SHA-256 Hash。

驗收條件：

* 每份 PDF 主檔有 metadata。
* PDF 主檔 Hash 可驗證。
* metadata 可追溯報表產生來源。

### M4-02 建立 PDF 儲存區

功能描述：

* PDF 實體檔保存於 NAS 或 Object Storage。
* 使用者不可直接存取 PDF 主檔路徑或 bucket。
* 支援保存年限與封存策略欄位。

驗收條件：

* PDF 主檔不可透過直接 URL 下載。
* 系統只能透過下載閘道提供副本。
* PDF 主檔不可覆蓋，只能新增版本或標記狀態。

### M4-03 建立 PDF 主檔完整性檢查

功能描述：

* 支援主檔 Hash 驗證。
* 支援背景 reconciliation 檢查 metadata 與實體檔案一致性。

驗收條件：

* PDF 不存在可被偵測。
* PDF Hash 不符會產生 Critical 告警。
* 修復流程有紀錄。

### M4-04 建立 PDF reconciliation 處理

功能描述：

* 偵測 orphan PDF。
* 偵測 missing metadata。
* 偵測 metadata 與實體檔案不一致。
* 建立修復、標記異常與人工覆核流程。

驗收條件：

* orphan PDF 可被列出。
* missing metadata 可被列出。
* 修復與人工覆核均有紀錄。

## 6. 里程碑 M5：下載閘道與浮水印

M5 屬於高資安風險里程碑，任務卡需納入 Security / Permission Agent、Audit / Evidence Agent、QA / Validation Agent 與 Reviewer Agent；下載授權、PDF library、簽章等級、極高機密是否可下載與 signed URL / token TTL 需依 ADR 或人類簽核結果執行。

### M5-01 建立下載閘道

功能描述：

* 所有下載必須經過後端授權檢查。
* 產生 Download ID / Serial No.
* 建立下載申請與下載紀錄。

驗收條件：

* 未授權使用者不可下載。
* 每次下載都有唯一序號。
* 下載紀錄包含使用者、時間、IP、User-Agent、報表版本。
* 下載閘道需由 Security / Permission Agent 與 Reviewer Agent 檢查未授權、跨部門、過期 token 與重放攻擊案例。

### M5-02 建立動態浮水印

功能描述：

* 下載副本加上下載人、帳號、部門、下載時間、來源 IP、下載序號、報表代號與版本。
* 支援查核碼；依機密等級與文件類型決定是否加 QR Code。
* QR Code 不應包含敏感明文，只能放查核碼或反查識別碼。

驗收條件：

* 每頁都有動態浮水印。
* 浮水印失敗不得提供未加浮水印 PDF。
* 外流文件可由下載序號、查核碼或 QR Code 反查。
* QR Code 是否啟用可由高機密 PDF 控制策略決定。
* 動態浮水印與查核碼需由 Audit / Evidence Agent 驗證可反查下載紀錄，且不得宣稱可 100% 防竄改。

### M5-03 建立下載副本 Hash

功能描述：

* 每次下載輸出後計算下載副本 Hash。
* 保存 Hash 與 Download ID 對應。

驗收條件：

* 下載副本 Hash 可查詢。
* 稽核人員可用下載序號反查下載紀錄。

### M5-03A 建立下載副本保存策略

功能描述：

* 預設只保存下載副本 Hash 與 metadata，不長期保存下載副本檔案。
* 若需保存下載副本，需經 Security Admin 核准。
* 保存下載副本需設定保存期限、加密與存取稽核。

驗收條件：

* 一般下載不留下長期副本檔案。
* 特殊保存案例有核准紀錄與到期策略。
* 稽核可查詢 Hash、Download ID 與保存狀態。

### M5-04 建立高機密 PDF 控制

功能描述：

* 依機密等級套用不同 PDF 控制。
* 一般內部報表：固定浮水印 + 動態浮水印 + 下載稽核。
* 機密報表：加 QR Code / 查核碼 + PDF 主檔 Hash + 下載副本 Hash。
* 高機密報表：加數位簽章 + timestamp + 局部 flattening。
* 極高機密報表：原則上不下載，只允許安全預覽、遮罩、限時、主管核准或 session recording。

驗收條件：

* 不宣稱 PDF 可 100% 防竄改、防截圖或防破解。
* 高機密報表下載需額外核准。
* 極高機密報表預設不可下載。
* 高機密與極高機密策略需取得人類或 ADR 簽核；若與資安或稽核要求衝突，任務卡不得 closure。

## 7. 里程碑 M6：查詢、預覽、覆核、重產與作廢

### M6-01 建立報表查詢

功能描述：

* 支援報表代號、報表名稱、資料日期、產生日期、批次編號、版本、狀態、機密等級、部門與交易人員查詢。
* 查詢結果依使用者 Data Scope 過濾。

驗收條件：

* 使用者只能看到授權範圍內報表。
* 查詢條件與查詢人員寫入稽核紀錄。

### M6-02 建立安全預覽

功能描述：

* 預覽需套用預覽浮水印。
* 高機密報表可禁止預覽或採遮罩、頁數限制、圖片化預覽。

驗收條件：

* 預覽不暴露 PDF 主檔路徑。
* 預覽紀錄寫入稽核。

### M6-03 建立覆核流程

功能描述：

* 報表發布前必須覆核。
* 覆核者可核准、退回或要求重產。
* 覆核需保存意見。

驗收條件：

* 報表作業員不得覆核自己產生或重產的報表。
* 覆核結果可追蹤。

### M6-04 建立重產流程

功能描述：

* 重產需填寫原因碼與說明。
* 重產產生新版本，不覆蓋舊版。
* 舊版標記已取代。

驗收條件：

* 新舊版本可比對。
* 重產申請人、執行人、覆核人可查詢。

### M6-05 建立作廢流程

功能描述：

* 已發布報表不可直接刪除，只能作廢。
* 作廢需申請與覆核。
* 已作廢報表不得被一般使用者下載。

驗收條件：

* 作廢紀錄可查詢。
* 稽核人員仍可查詢作廢歷史。

---

## 8. 里程碑 M7：稽核、Log、告警

M7 屬於稽核與證據能力核心里程碑，任務卡需指定 Audit / Evidence Agent 與 Reviewer Agent；fail-closed 邊界、hash chain 範圍、WORM / 獨立稽核 DB 與告警通報責任需依 ADR 或人類簽核結果執行。

### M7-01 建立稽核查詢與 Hash Chain

功能描述：

* 延伸 M1-04 的稽核寫入基礎。
* 建立稽核查詢、稽核匯出與查詢遮罩。
* 建立 Hash Chain 或週期性 root hash。
* 稽核查詢本身也需被稽核。

驗收條件：

* 敏感操作可被稽核查詢。
* 稽核匯出有權限限制與稽核紀錄。
* Hash Chain 或 root hash 可驗證稽核紀錄未被任意竄改。
* 稽核鏈設計需經 Audit / Evidence Agent review，涉及 fail-closed 或 WORM 時需 human / ADR gate。

### M7-02 建立 Log 敏感資料防護

功能描述：

* Log 不得保存密碼、Token、Session ID 原文、connection string、private key、API key、完整身分證號或完整金融帳號。

驗收條件：

* Log 掃描無敏感欄位。
* 錯誤摘要不暴露敏感設定。

### M7-03 建立告警分級

功能描述：

* Critical：高機密報表未授權下載、PDF 主檔 Hash 不符、稽核寫入失敗、批次重大錯誤、break-glass 使用。
* Error：報表產生失敗、匯入失敗、下載閘道錯誤、權限同步失敗。
* Warning：異常下載頻率、資料差異偏高、檢核警示。
* Info：一般查詢、一般下載、一般排程完成。

驗收條件：

* Critical 事件 5 分鐘內通報。
* Error 事件產生 Email 或工單。
* Warning 事件進入每日報表。
* 告警分級、通報對象與內部群組整合需有 owner review 與營運簽核。

---

## 9. 里程碑 M8：Admin 與 Break-glass

M8 屬於高權限治理里程碑，任務卡需指定 Security / Permission Agent 與 Reviewer Agent；Admin 職責分離、自訂角色、Data Scope、dual-control、MFA 與 break-glass 權限範圍需依 ADR 或人類簽核結果執行。

### M8-01 建立角色管理

功能描述：

* Admin 可建立、複製、停用與修改角色。
* Admin 可指派角色權限。
* Admin 可對使用者或 AD 群組指派角色。
* 角色管理需沿用 M1-04 稽核基礎與 M2-04 Data Scope 基礎規則。

驗收條件：

* 角色異動有申請、核准、生效時間與稽核紀錄。
* Admin 不得自行授權取得未經核准的高風險權限。
* 角色建立、角色授權與高風險權限異動需經 Security / Permission Agent review，必要時需雙人覆核。

### M8-02 建立資料範圍管理

功能描述：

* Admin 可設定使用者、角色或 AD 群組的 Data Scope。
* 支援部門、分公司、報表代號、機密等級與資料日期範圍。

驗收條件：

* 權限測試可驗證授權與未授權案例。
* Data Scope 異動寫入稽核。
* Data Scope 規則需由 reviewer 驗證不會因 Admin 權限自動取得報表內容權限。

### M8-03 建立 break-glass 流程

功能描述：

* break-glass 帳號需事前核准、限時啟用、使用後自動失效。
* 使用時需填寫原因、事件編號與預期處理範圍。
* 啟用需 dual-control 核准。
* 使用時需通過 MFA / OTP 或等效強驗證。
* 權限需以限時、限範圍 token 或臨時角色授權。
* 所有 SQL、shell command、檔案操作、來源 IP、時間、結果都需稽核。

驗收條件：

* break-glass 使用產生 Critical 通報。
* break-glass 權限不可超出核准範圍。
* break-glass token 到期後自動失效。
* 使用後需由稽核或主管覆核操作內容。
* PAM vault、限時帳號與 session recording 列為 Level 2 或高機密場景決策項，不作為第一版一般報表硬需求。
* break-glass 啟用、使用與事後覆核需 human sign-off；若 runtime 阻擋不足，必須以人工 review 與 command-backed evidence 補位。

## 10. 里程碑 M9：遷移驗證與切換

M9 屬於正式切換與舊系統風險里程碑，任務卡需指定 Captain / Coordinator、QA / Validation Agent、Security / Permission Agent、Audit / Evidence Agent 與 Reviewer Agent；DB 選型、Shadow DB 升格、Pilot 範圍、Go / No-Go、rollback window 與舊系統下線需人類或 ADR 簽核。

### M9-01 建立 Shadow DB

功能描述：

* 保留 MSSQL 作為正式 leader。
* 建立經 PoC 選定的 Shadow DB / follower。
* 透過 CDC、ETL、Outbox 或批次同步將資料寫入 Shadow DB。
* Shadow DB 只做比對與驗證，不影響正式報表結果。

驗收條件：

* 同步錯誤進入 DLQ 或錯誤清單。
* Shadow DB 不影響正式報表產出。
* Shadow DB 的網段隔離、DB 權限、加密、備份、資料保留、存取稽核與 DBA 責任已確認。
* Shadow DB / follower 的 DB 選型與同步策略需完成 ADR 簽核後才可進入正式切換 gate。

### M9-02 建立 Go / No-Go Gate

功能描述：

* 建立正式切換前檢查項目。
* 包含資料比對、報表比對、權限測試、稽核測試、PDF 安全、NFR、效能測試、DR 演練與人工簽核。

驗收條件：

* Gate 未通過不得正式切換。
* 每個 Gate 都有證據與簽核紀錄。
* Go / No-Go 必須包含 reviewer 結論、human sign-off、ADR 狀態與 rollback readiness。

### M9-03 建立 Rollback Runbook

功能描述：

* 建立切換失敗時回復舊系統的步驟。
* 記錄 DNS / routing、資料同步、排程、報表產生、下載閘道與通知回復方式。

驗收條件：

* 完成演練。
* RTO / RPO 達標。

### M9-04 建立 UAT、訓練與操作手冊

功能描述：

* 建立業務、稽核、資安與維運 UAT 腳本。
* 建立系統操作手冊、異常處理手冊與值班通報流程。
* 完成使用者訓練與簽到紀錄。

驗收條件：

* UAT 缺陷均有處理狀態。
* 操作手冊與異常處理手冊完成。
* 業務、稽核、資安與維運完成簽核。
* UAT 與訓練結果需由 QA / Validation Agent 彙整，缺陷關閉與例外接受需有 owner 簽核。

### M9-05 建立 Pilot 與舊系統下線 Gate

功能描述：

* 先以非正式報表或低機密報表進行 Pilot。
* 建立分批、分部門或分報表等級切換計畫。
* 建立 MSSQL leader 下線條件。

驗收條件：

* Pilot 問題均有處置紀錄。
* Rollback Window 期間可回復舊系統。
* 連續 N 批次或 N 天雙製比對通過。
* MSSQL leader 下線需取得業務、稽核、資安與維運簽核。
* Pilot 擴大、正式切換或舊系統下線不得由 Agent Team 自動決定，必須 human / ADR gate 通過。

## 11. 里程碑 M10：第一版上線驗收

### M10-01 功能驗收

驗收條件：

* 舊系統必要報表範圍已覆蓋。
* Golden Dataset 全案例與 expected result 比對通過。
* 新舊報表雙製比對通過。
* PDF 主檔 Hash 與下載副本 Hash 可追蹤。
* 未授權使用者無法查詢、預覽、下載報表。
* 高機密報表下載需通過額外控管。
* 稽核紀錄完整，且稽核查詢本身有紀錄。
* Critical 事件能在 5 分鐘內通報。
* 批次重跑不會產生重複資料。
* 報表重產不覆蓋舊版。
* 作廢流程需覆核並保留歷史。
* NFR、RTO、RPO、批次完成時間、下載延遲與錯誤率達標或有核准例外。
* 備份還原與 rollback runbook 演練通過。

### M10-02 舊系統覆蓋確認

驗收條件：

* 每份舊系統正式報表都有新系統對應項。
* 每個舊系統資料來源都有承接方式。
* 每個舊系統權限群組都有新角色或 Data Scope 對應。
* 每個舊系統排程或手動作業都有新流程對應。
* 被廢止的舊功能需有業務或稽核核准。
* 舊系統下線條件已達成，或保留並行運轉期間與責任人。
