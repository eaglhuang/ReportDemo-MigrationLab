## 1. 文件目的

本文件是 ReportDemo Migration Lab 的演練總入口，用來說明如何以 Qutora 作為本演練舊系統，逐步驗證 PDF 資料移轉、資料庫轉換、下載閘道、動態浮水印、權限、稽核、平行作業、Go / No-Go 與 rollback。

本文件是 execution layer，不取代既有三份主文件：

- 架構與治理以 `內部人員交易報表轉媒體儲存系統_系統架構與治理計畫書.md` 為準。
- 功能範圍與任務卡以 `內部人員交易報表轉媒體儲存系統_功能里程碑計畫.md` 與 `tasks/` 為準。
- 人類決策與簽核以 `決策紀錄樣板ADR.md` 為準。

## 2. 本演練舊系統

本演練舊系統採用 Qutora：

- 原始碼位置：`open-source-sandbox/qutora-api`
- 固定 commit：`de156e0eb72d58772a76e570eb711db344bedfc0`
- 啟動基準：`open-source-sandbox/qutora-api/samples/docker-compose.sqlserver.yml`
- 初始資料庫：Qutora SQL Server / `QutoraDB`

Qutora 在本演練中代表既有文件與媒體管理系統，負責提供文件上傳、metadata、分類、下載、SQL Server DB、基礎權限與 audit log。所有搬移演練、比對與新系統設計都以 Qutora 作為舊系統來源。

邊界聲明：Qutora 代表本次演練舊系統，不代表真實券商內部人員交易報表舊系統的完整業務規則、正式資料、法遵規則或報表計算邏輯。

## 3. 分階段策略

| 階段 | 目標 | 文件 |
| --- | --- | --- |
| MVP1 | 2 週內驗證 Qutora 舊系統、PDF / metadata / DB / audit baseline 是否可跑通。 | `drills/MVP1兩週風險驗證計畫.md` |
| MVP2 | 根據 MVP1 結果調整策略，驗證 MariaDB 初版、下載閘道 PoC、動態浮水印 PoC 與第一批 validator。 | 後續建立 |
| Pilot | 擴大到批次 PDF 移轉、新舊資料比對、權限、稽核、警示與平行作業。 | 後續建立 |
| Production Candidate | 驗證 Go / No-Go、正式切換、備份、restore、rollback、break-glass 與緊急維修。 | 後續建立 |

12 週不是 MVP 週期，而是從 MVP1、MVP2、Pilot 到 Production Candidate 的壓縮目標。實際時程需依 Qutora 啟動結果、PDF 量級、MariaDB 改造難度、驗證缺口與 Go / No-Go evidence 每 2 週重新估算。

## 4. 三人小隊假設

| 角色 | 投入比例 | 主要責任 |
| --- | ---: | --- |
| Tech Lead / Captain | 60% | 任務拆解、風險控管、ADR、review routing、Go / No-Go evidence。 |
| Backend / DBA | 100% | Qutora DB 盤點、MariaDB 轉換、API / migration / validator、資料一致性。 |
| QA / Security / DevOps | 80% | Golden Dataset、權限與稽核驗證、Docker、CI evidence、rollback runbook。 |

三人皆假設已有 AI 協作環境，可立即閱讀計畫、執行任務卡、留下 command-backed evidence。AI 可輔助分析與產出，但資安、稽核、資料、正式切換與 ADR 仍需人類簽核。

## 5. 任務卡落地原則

任務卡採由大到小細化。先完成演練總計畫與 MVP1 / MVP2 計畫，再升級核心任務卡；非核心任務卡保留摘要格式，但未補齊完整設計規格前不得開工。

完整任務卡必須包含：真實功能場景、API / 資料表 / 狀態機 / 錯誤碼 / 權限 / 稽核 / fail-closed、影響範圍、輸入輸出、完成定義、10 條 validators、10 條 test cases、風險回復、reviewer / human gate / ADR。
