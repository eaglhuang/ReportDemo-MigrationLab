# ReportDemo 每日任務卡排程

版本：v1.0  
狀態：Active  
適用情境：三人皆已安裝 AI 開發環境，每人每日最多 8 小時，每週 5 個工作日，週末不排正式工作。

## 1. 文件目的

本文件只維護每日 WHAT/WHEN roster：哪一天、哪個角色、對應哪張任務卡、預期產出什麼 evidence。

每日 HOW 不在本文件重複維護。派工單格式、每日六步驟、AI Work Report、review checklist、EOD 與週末規則全部依 `runbooks/RB-06-ai-dispatch-cycle.md`。

## 2. 使用方式

1. 找到今天的 Week / Day 列。
2. 依該列確認三個角色的任務卡與預期 evidence。
3. 回到 RB-06 建立或更新 `evidence/<Stage>/daily-dispatch-YYYY-MM-DD.md`。
4. 讀今日任務卡；`primary_role` 是 DRI，`closure_reviewer` 是關卡 reviewer。
5. 若只支援某張卡的局部工作，派工單用 `TASK-RPT-0004 / 0004b contributor` 這類格式標示。

## 3. 與 RB-06 的分工

| 文件 | 負責內容 |
| --- | --- |
| `drills/每日任務卡排程.md` | WHAT/WHEN roster、週期安排、每日預期產物。 |
| `runbooks/RB-06-ai-dispatch-cycle.md` | HOW、派工單格式、每日六步驟、review、EOD、週末規則。 |
| `tasks/TASK-RPT-*.task.md` | 單張任務的 DRI、reviewer、scope、deliverables、validators、evidence path。 |

例外處理規則：

- 上表 9 項在 W14 的 release acceptance（`0044`）與 legacy coverage confirmation（`0045`）中記為 **documented exception**，不算 blocking finding，不阻擋演練封版。

- 全部 9 項需彙整進 W14D5 的 next-phase recommendation，作為正式專案的待辦輸入。
- **0027 特別規則**：高機密 PDF 控制在「資料安全最高優先」原則下被裁減，必須是顯式人類決策——W12D4 代理簽核演練時，Tech Lead / Captain 與 Security proxy 需在 `sign-off-record.md` 的 `conditions` 欄記載「本演練未驗證高機密 / 極高機密控制，正式專案上線前必須先完成 ADR-007 與 TASK-RPT-0027」，缺此記載不得做出 conditional-go 以上的結論。
- 若演練期間發現任一被裁減項目實際阻擋了其他卡的驗證（例如 Pilot 高機密樣本測試需要 0027 的策略），依 §9 重排規則升級 Tech Lead 裁決：補升級該卡或調整測試範圍，二擇一並記入 ChangeLog。
