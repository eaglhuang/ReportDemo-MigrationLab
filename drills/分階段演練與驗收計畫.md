## 1. 演練總則

本文件是 ReportDemo Migration Lab 的唯一演練執行指南，整合原本的演練總計畫、MVP1、MVP2、Pilot、Production Candidate 與 Gate 文件。

本演練舊系統採用 Qutora：

| 項目 | 內容 |
| --- | --- |
| repo path | `open-source-sandbox/qutora-api` |
| 固定 commit | `de156e0eb72d58772a76e570eb711db344bedfc0` |
| 啟動方式 | `open-source-sandbox/qutora-api/samples/docker-compose.sqlserver.yml` |
| 舊系統 DB | Qutora SQL Server / `QutoraDB` |
| 目標 DB | MariaDB，依 ADR-013 |

邊界聲明：Qutora 代表本次演練舊系統，用於驗證搬移流程、資料移轉、下載、權限、稽核與平行驗證；不宣稱涵蓋真實券商舊系統全部業務規則、正式資料、法遵規則或報表計算邏輯。

## 2. 階段策略與時程

2 週 MVP 是風險驗證節奏，不是完整上線承諾。12 週是壓縮目標；合理預估應採 Base / Expected / Risk Buffer 三段重新估算。若三人皆具 AI 開發環境，且由 AI 主開發、人類監控決策與驗收數據，壓縮排程以 `drills/AI主導三人併行排程與缺口分析.md` 為準。

| Stage | 建議週期 | 目的 | 進入條件 | 退出 Gate |
| --- | ---: | --- | --- | --- |
| MVP1 | 2 週 | 驗證 Qutora 可啟動、可操作、可取得 PDF / metadata / DB / audit baseline。 | submodule 與 Docker 可用。 | Qutora baseline evidence 完整，且人類接受限制。 |
| MVP2 | 2 週 | 驗證 MariaDB 初版轉換、下載閘道 PoC、動態浮水印 PoC 與第一批 validator。 | MVP1 Gate 通過。 | 下載、浮水印、hash、audit fail-closed 均可證明。 |
| Pilot | 4 到 8 週 | 擴大到批次移轉、新舊資料比對、權限、稽核、警示與平行作業。 | MVP2 Gate 通過且 Pilot 核心卡升級完成。 | 平行作業差異可解釋，重大問題有處置。 |
| Production Candidate | 4 到 8 週 | 建立 Go / No-Go package、備份還原、rollback、break-glass 與正式切換 evidence。 | Pilot Gate 通過。 | Go / No-Go 證據齊全且 human sign-off 完成。 |

### 2.0 AI 主導排程補充

AI 主導情境採下列額外前提：

- `[AI]` AI 主導產出程式、腳本、文件草稿、validator、測試與 evidence 草稿。
- `[AI->HUMAN]` AI 先產出，人類 review 後才算完成。
- `[HUMAN]` 人類主導 ADR、資安 / 稽核 / 權限例外、Go / No-Go、正式簽核與驗收數據接受。
- `[GATE]` 階段阻擋條件與 closure 判斷。
- 每人每日最多 8 小時、每週 5 個工作日、週末不排正式工作；週末 AI 草稿不得未經下一工作日 review 就納入 evidence。

在此情境下，Base Plan 為 12 到 14 週完成 Production Candidate 演練；Best Case 8 到 10 週僅作挑戰目標；若 W3 結束仍無法證明下載閘道、浮水印、hash 與 audit fail-closed，需退回 16 到 18 週保守排程。

## 4. 三人小隊執行與驗收模型

| 角色 | 投入比例 | 主責 | 驗收責任 |
| --- | ---: | --- | --- |
| Tech Lead / Captain | 60% | 範圍控制、任務排序、ADR / Gate、review routing、Go / No-Go evidence 彙整。 | 不驗收自己產出的 evidence；負責確認 Gate 是否可交人類簽核。 |
| Backend / DBA | 100% | Qutora DB 盤點、MariaDB schema、移轉腳本、下載 / 浮水印 PoC、資料一致性 validator。 | 接受 QA / Security / DevOps 複核。 |
| QA / Security / DevOps | 80% | Docker、合成資料、evidence、權限負向測試、audit / rollback / fail-closed 驗證。 | 對 Gate 具 blocking 權限。 |

代理簽核原則：演練可由三人小隊代理產出技術 evidence，但業務、資安、稽核、正式切換與舊系統下線不得由 Agent 或單一工程角色自行簽核。

### 4.1 每階段責任矩陣

| Stage | Tech Lead / Captain | Backend / DBA | QA / Security / DevOps | Gate reviewer | 不得自審規則 |
| --- | --- | --- | --- | --- | --- |
| MVP1 | 排定啟動順序、確認 Qutora commit、裁定 blocking issue。 | 協助 DB / API 盤點與 Qutora 技術問題定位。 | 主責 RB-01、RB-02、RB-03 evidence、啟動驗證與負向測試。 | Tech Lead / Captain + QA / Security / DevOps 交叉驗收。 | Qutora 啟動 evidence 由 QA / DevOps 產出時，需 Tech Lead 或 Backend 複核。 |
| MVP2 | 控制 MVP2 scope、確認 MariaDB / 下載 / 浮水印 PoC 是否可進 Pilot。 | 主責 staging、metadata、下載閘道、浮水印、Hash 設計。 | 主責權限負向測試、audit、fail-closed、evidence package。 | QA / Security / DevOps 對 Backend 產出具 blocking 權限。 | Backend / DBA 不得單獨驗收自己產出的 schema、API 或 Hash 結果。 |
| Pilot | 彙整平行作業與 Gate evidence，決定是否提送人類簽核。 | 支援批次移轉、查詢效能、資料修復與 production-like 問題定位。 | 主責雙製比對、reconciliation、告警、權限、稽核與 rollback evidence。 | Tech Lead / Captain + QA / Security / DevOps；稽核類 evidence 需 Audit / Evidence role 複核。 | 產出 reconciliation / audit evidence 的角色不得單獨標記 Gate pass。 |

### 4.2 代理簽核表

演練期間若沒有真實業務、稽核、資安或維運代表，需先指定代理人；代理人只代表演練簽核，不代表真實券商正式簽核。

| 簽核角色 | 演練代理人 | 可簽核範圍 | 不可簽核範圍 | Evidence 欄位 |
| --- | --- | --- | --- | --- |
| Business Owner | Tech Lead / Captain | MVP1/MVP2/Pilot 演練範圍、功能覆蓋建議、非正式驗收意見。 | 正式業務上線、真實券商功能裁撤、正式報表責任歸屬。 | `business_proxy`, `business_signed_at`, `business_decision` |
| Audit Owner | QA / Security / DevOps 或指定 Audit / Evidence reviewer | evidence 完整性、稽核事件、Hash、Gate package 是否可追溯。 | 真實法遵保存年限、正式稽核例外、監理承諾。 | `audit_proxy`, `audit_signed_at`, `audit_decision` |
| Security Owner | QA / Security / DevOps | 權限負向測試、fail-closed、break-glass 演練、敏感資料遮罩。 | 正式資安風險接受、MFA / PAM 採購、正式資安例外放行。 | `security_proxy`, `security_signed_at`, `security_decision` |
| Operations Owner | Backend / DBA 或 QA / Security / DevOps | Qutora 啟動、Docker、DB、rollback dry run、RTO/RPO 演練紀錄。 | 正式值班責任、正式維運 SLA、正式災難復原承諾。 | `ops_proxy`, `ops_signed_at`, `ops_decision` |
| Go / No-Go Approver | 不得由單一代理人決定 | 只能彙整建議與 evidence readiness。 | 正式 Go / No-Go、舊系統下線、長期並行結束。 | `final_decision_owner`, `final_decision`, `adr_refs` |

### 4.3 Gate Review 規則

- 任務卡產出者不得單獨驗收自己的 deliverables 或 evidence。
- 若 reviewer 與 producer 因三人小隊限制必須重疊，需在 evidence index 標記 `review_conflict=true`，並由 Tech Lead / Captain 補做第二層 review。
- Gate review 若涉及資安、稽核、資料一致性、正式切換或舊系統下線，三人小隊只能提出建議，不得取代 human / ADR gate。
- `ProductionCandidate` 階段的正式簽核紀錄需使用 `runbooks/RB-03-evidence-standard.md` 的 sign-off record template。

### 4.4 CI 與命令型 Evidence 定位

本演練第一版不要求先建立 GitHub Actions 或企業 CI。可重跑的本機命令、腳本輸出、hash、DB dump / restore 紀錄與 reviewer 重跑結果，視為本演練的正式 command-backed evidence。若後續新增 CI，CI artifact 應補充 evidence，而不是取代 RB-03 的 evidence index、Gate summary 與人類 / ADR gate。

## 5. MVP1：兩週風險驗證

目標是在 2 週內確認 Qutora 可被穩定啟動、操作、盤點與作為後續搬移基準。MVP1 不追求完整新系統功能，只驗證舊系統來源、PDF、metadata、DB、下載、權限與 audit evidence 是否可用。

### 5.1 Qutora 能力限制與 Fallback 規則

若 Qutora 的實際功能與本演練需求不同，不能直接修改 Qutora 原始碼補齊，也不能把缺口偽裝成已完成。處理順序如下：

1. 先用 Qutora 原生 API / DB / log 證明可取得的能力。
2. 若缺少報表代號、機密等級、audit 欄位或權限欄位，使用 `tools/generate_synthetic_pdf.py` 與 `RB-02` 產出的合成 metadata 作為演練補充資料。
3. 在 evidence index 標記 `qutora_capability_gap=true`、缺口描述、fallback 來源與 reviewer。
4. 只要缺口影響資料一致性、權限、稽核或正式切換判斷，必須列為 Gate blocking 或 ADR / 人類簽核項。
5. fallback 只能支援演練，不得宣稱 Qutora 已具備該能力，也不得代表真實券商舊系統已覆蓋。

## 9. Gate 總表

| Gate | 不得前進的條件 |
| --- | --- |
| MVP1 -> MVP2 | Qutora 不可重複啟動、PDF / metadata / DB / audit 任一基準不可取得、evidence 不可重跑。 |
| MVP2 -> Pilot | MariaDB mapping 不可信、下載閘道可被繞過、浮水印失敗會回傳未處理 PDF、audit fail-closed 不成立。 |
| Pilot -> Production Candidate | 平行作業差異未分類、權限越權、稽核缺口未處置、告警路由不明。 |
| Production Candidate -> Go | backup / restore / rollback 未通過、human sign-off 不完整、舊系統回復路徑不明。 |
